# One country, multiple portraits: representativeness in GPS-based mobility data is source-specific and spatially dependent

This repository contains the data, code, model outputs, and supporting documentation for the paper:

> One country, multiple portraits: representativeness in GPS-based mobility data is source-specific and spatially dependent

The repository is organized as a journal companion archive. It is intended to let readers inspect the municipal-level data inputs, rerun the main computational workflows, and trace the outputs used to assemble the paper figures and tables.

## Repository Structure

```text
data/
  raw/                         Municipal-level input data and spatial files
  processed/                   Model-ready inputs for the explain-bias workflows
code/
  01_measure_bias/             Workflow to estimate representativeness bias
  02_explain_bias/             Holdout and blocked cross-validation modelling workflows
  03_figures_tables/           Reserved for figure/table assembly helpers
  style/                       Shared plotting styles
src/
  MeasureBiasV.0.0/            Reusable R source module for bias measurement
  ExplainBiasV.0.0/            Reusable R source module for bias explanation models
  *.tar.gz                     Archived source-package snapshots
results/
  measure_bias/                Outputs from the bias measurement stage
  explain_bias/                Model outputs, predictions, SHAP values, maps, and diagnostics
  plots/                       Additional output plots, not final manuscript figures
manuscript/
  figures/                     Figure assets, code, intermediate data, and plots
  supplementary-figures/       Supplementary figure assets
  tables/                      Manually assembled table artifacts
docs/
  methods/                     Method notes and hyperparameter documentation
  project_notes/               Historical project notes retained for transparency
  variable_dictionary/         Variable documentation
```

Historical exploratory results are intentionally excluded from this journal-facing branch.

## Data

The raw data included here are aggregated at the municipality level. They include:

- census-derived municipal population data;
- aggregated active-population counts from Facebook mobility data;
- aggregated active-device counts from phone/GPS-based mobility data;
- spatial boundary files used for municipal and state-level analyses.

See [data-availability.md](data-availability.md) for the data and code availability statement.

## Reproducibility Workflow

The workflows are written in R/Quarto. The main computational path is:

1. Run [code/01_measure_bias/measure_bias_full.qmd](code/01_measure_bias/measure_bias_full.qmd) to compute representativeness bias from raw municipal population and active-population inputs.
2. Use the processed explain-bias inputs in [data/processed/explain_bias](data/processed/explain_bias) for the modelling workflows.
3. Run the explain-bias workflows in [code/02_explain_bias](code/02_explain_bias):
   - `explain_bias_holdout_gridsearch_pvalue.qmd`
   - `explain_bias_blocked_cv_gridsearch.qmd`
   - `explain_bias_blocked_cv_randomsearch.qmd`
4. Inspect model outputs under [results/explain_bias](results/explain_bias).
5. Inspect manuscript figure assets and manually assembled table artifacts under [manuscript](manuscript).

The final paper figures and tables were assembled manually from outputs generated by the scripts. The underlying computational outputs and supporting plot/table assets are included.

## Software Requirements

The analysis uses R and several R packages, including:

- `tidyverse`, `dplyr`, `readr`
- `sf`, `spdep`, `spgwr`
- `ggplot2`, `viridis`, `patchwork`
- `xgboost`, `caret`, `yardstick`, `Metrics`
- `SHAPforxgboost`, `shapviz`, `pdp`
- `foreach`, `doParallel`, `parallel`

The Quarto files currently install missing packages automatically when run. For long-term reproducibility, a future improvement would be to add an `renv.lock` file.

The [src](src) directory contains reusable R source modules from the project. The `.tar.gz` files in that directory are built source-package archives of those modules, retained as snapshots for transparency; the primary entry points for reproducing the paper are the workflows in [code](code).
